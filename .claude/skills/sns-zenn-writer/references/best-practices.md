## 1. アルゴリズムの仕組み（2025-2026年版）

### 1.1 トレンドランキングの仕組み

Zenn のトレンドは公式にアルゴリズムの詳細を公開していないが、コミュニティの分析から以下が判明している:

- **主要シグナル**: いいね（Like）数の増加速度が最重要
- **鮮度**: 新しい記事ほどトレンドに載りやすい
- **閲覧数**: 直接的なランキング要因ではないが、いいねの母数として間接的に影響
- **ブックマーク**: いいねと同様にエンゲージメント指標として評価される

Source: [Zennで自分の記事の閲覧数を確認する方法](https://zenn.dev/pike/articles/5daea8b714e519), [JuliaでZennをスクレイピングしてランキングを作ったよ](https://zenn.dev/ohno/articles/098dbf3729b431)

### 1.2 トレンド入りの目安

Zenn は Qiita のように明確なトレンド入り条件を公開していないが、実績データから以下が推定される:

| 段階         | いいね数の目安 | 効果                                     |
| ------------ | -------------- | ---------------------------------------- |
| 注目記事     | 10-20 いいね   | トレンドページ下位に表示される可能性     |
| トレンド入り | 30-50 いいね   | トレンドページに安定表示、自然流入が急増 |
| バズ記事     | 100+ いいね    | トップページ上位、SNS でも拡散           |
| 殿堂入り     | 500+ いいね    | 長期的にトレンドページに残る             |

**重要**: いいねの「速度」が絶対数より重要。投稿直後24時間のいいね獲得速度がトレンド表示を左右する。

**参考統計（220記事投稿者の実績）:**

- 平均いいね数: 約10
- 平均閲覧数: 1,923回/記事
- 最大閲覧数: 28,727回/記事
- 閲覧数の分布は非常にばらつきが大きい（一部の記事がPVの大半を稼ぐ）

Source: [この1年ほぼ毎日Zenn記事を書き続けて思ったこと](https://zenn.dev/acntechjp/articles/8a4547ea504f48)

### 1.3 エンゲージメント指標の優先度

| シグナル                | 重要度 | 備考                                       |
| ----------------------- | ------ | ------------------------------------------ |
| いいね速度（投稿後24h） | 最高   | トレンド表示の最大要因                     |
| いいね総数              | 高い   | トレンド維持に影響                         |
| ブックマーク            | 高い   | 「後で読む」は記事の実用性の証明           |
| コメント                | 中程度 | 会話が発生する記事は価値が高いと評価される |
| 閲覧数                  | 間接的 | いいねの母数として間接的に寄与             |
| SNS共有                 | 間接的 | 外部からの流入がいいね数を押し上げる       |

### 1.4 「第二波」現象

人気記事は公開直後に急速に閲覧数が増加し、その後も長期間安定した閲覧を獲得する。特筆すべきは、一部の記事が**公開から数週間後に再び閲覧数が増加する「第二波」**を経験すること:

- **原因**: Google検索インデックス化、SNSでの再シェア、関連記事からのリンク
- **対策**: 検索に強いタイトル設計が第二波を生みやすい

Source: [2025年振り返り 自分のZenn記事48本をAIに分析させてみた](https://zenn.dev/acntechjp/articles/f40e7f3c4f671b)

### 1.5 SEO 特性

Zenn のドメインパワーは着実に成長しており、Google 検索で以下の特性がある:

- **ドメインの強さ**: zenn.dev は技術系キーワードで上位表示されやすい
- **長文記事に強い**: 体系的な解説記事は検索評価が高い
- **タグページ**: `/topics/{tag}` がカテゴリページとして検索に露出する
- **外部リンクが重要**: SNS や他ブログからのリンク獲得が最優先のSEO施策

Source: [ZennのSEOについて考察してみた](https://zenn.dev/icedog410/articles/fb1cb96e576a8ea8e202)

## 2. プラットフォーム特性

### 2.1 基本情報

- **運営**: Classmethod 傘下（2023年買収）
- **対象**: 日本語圏のソフトウェアエンジニア
- **特徴**: GitHub 連携で記事管理、「本」機能で有料販売、Publication で組織メディア運営
- **収益化**: 本の有料販売 + 読者からの有料バッジ（チップ）

### 2.2 コンテンツ形式

| 形式                     | 説明                                             | 収益化                    |
| ------------------------ | ------------------------------------------------ | ------------------------- |
| **Articles（記事）**     | 単体の技術記事。`type: tech` または `type: idea` | 不可（バッジのみ）        |
| **Books（本）**          | 複数チャプターで構成される体系的コンテンツ       | 有料販売可能（0-5,000円） |
| **Scraps（スクラップ）** | メモ・議論用のスレッド形式                       | 不可                      |

### 2.3 フロントマター仕様

```yaml
---
title: "記事タイトル（技術名 + 具体的内容）"
emoji: "🤖" # アイキャッチ用絵文字（必須）
type: "tech" # "tech" or "idea"
topics: ["claudecode", "typescript", "ai"] # 最大5個
published: true # false = 下書き
publication_name: "" # Publication に紐付ける場合のみ
---
```

### 2.4 スラッグ規則

- **文字数**: 12-50文字
- **使用可能文字**: 小文字英数字とハイフンのみ (`a-z0-9-`)
- **例**: `claude-code-agent-sdk-guide`
- **末尾に日付を付ける慣例**: `nextjs-tutorial-20260216`（衝突回避に有効）

### 2.5 記事タイプの違い

| タイプ | 用途                       | トレンド入りやすさ                 |
| ------ | -------------------------- | ---------------------------------- |
| `tech` | 技術的な知見の共有         | 高い（全体の90%以上を占める）      |
| `idea` | 技術以外のアイデア・ポエム | やや低い（ニッチな共感を得やすい） |

Source: [Zenn公式 - GitHubリポジトリ連携](https://zenn.dev/zenn/articles/connect-to-github)

## 3. コンテンツ方針

### 3.1 バズりやすい記事タイプ（優先度順）

1. **ファーストインプレッション** — 新ツール/ライブラリの速報記事（最もバズる）
   - 新モデル発表時に「出たらとりあえず触って書く」が最強戦略
   - 公開直後24時間が勝負
2. **チュートリアル** — 手順付きの実践ガイド
   - 長寿命記事になりやすい（第二波が期待できる）
3. **比較記事** — A vs B の実測データ付き比較
   - 「Cursor vs Claude Code」のような対立軸が明確な記事
4. **ディープダイブ** — 内部実装や仕組みの解説
   - 技術的に深い記事は権威付けに有効
5. **シリーズ記事** — 連載形式での継続的な発信
   - 「Claude Codeチャレンジ日誌」のような連載は読者追従性が高い

### 3.2 記事構成のベストプラクティス

- **導入**: 何を解決するか/なぜ重要かを3行以内で説明
- **環境・前提条件**: 使用技術のバージョン、OS などを明記
- **本論**: H2/H3 で論理的に区切る（5-8セクションが目安）
- **コードブロック**: **実行可能な完全版**を掲載（言語指定必須）
- **図解**: スクリーンショットや図を適宜挿入（わかりやすい記事の必須条件）
- **まとめ**: 要点の振り返り + 次のステップ

### 3.3 一次情報の価値

Zenn のコミュニティガイドラインは「実際に試した際の気づきや自身の考察を含めること」を推奨している。以下が一次情報として高く評価される:

- エラー解決の具体的な過程
- 技術導入の成功・失敗事例
- 独自のベストプラクティスの体系化
- 実測データに基づくパフォーマンス比較

### 3.4 リリース連動戦略

新ツール/ライブラリのリリース時に3段構成で最大効果を狙う:

```
Day 0: 速報記事（ファーストインプレッション）→ トレンド入り狙い
Day 1: 深掘り記事（チュートリアル or ディープダイブ）→ 実用的な情報提供
Day 7: まとめ記事（比較 or ベストプラクティス）→ 検索流入の受け皿
```

Source: [トレンド入りする記事を書く技術](https://zenn.dev/collabostyle/articles/858875b235cdd6)

## 4. タイトル設計

### 4.1 基本原則

- **技術名を必ず含める**: 「Claude Code」「Next.js」「MCP」などの固有名詞
- **70文字以内**: Zenn の表示上限と検索結果での見切れを考慮
- **具体的な数字を入れる**: 「秘訣4選」「3つの方法」は具体性と説得力を増す
- **具体的な動詞を使う**: 「始める」「解説する」「移行する」

### 4.2 高パフォーマンスタイトルパターン

| パターン            | 例                                             | 特徴             |
| ------------------- | ---------------------------------------------- | ---------------- |
| 技術名 + アクション | 「Claude Code で TDD を始める方法」            | 検索に強い       |
| 技術名 + 感想       | 「Claude Agent SDK v0.3 を試してみた」         | 速報向き         |
| A vs B              | 「Cursor vs Claude Code: 1週間使い比べた結果」 | 比較需要を獲得   |
| 完全解説            | 「MCP プロトコルの仕組みを完全解説」           | 権威性を主張     |
| 数字入り            | 「Next.js 16 の新機能 5 選」                   | クリック率が高い |
| 問題解決            | 「TypeScript の型エラーを3分で解決する方法」   | 長寿命           |

### 4.3 避けるべきタイトル

- **クリックベイト**: 「衝撃！」「知らないとヤバい」（ガイドライン違反リスク）
- **曖昧なタイトル**: 「メモ」「雑記」（検索に引っかからない）
- **長すぎるタイトル**: 80文字超（切れて意味不明になる）

Source: [Zennの記事タイトルがいいね数にどのくらい影響するのか テキストマイニングで分析](https://zenn.dev/sdb_blog/articles/98989898998989)

## 5. タグ（topics）設計

### 5.1 タグの役割

Zenn のタグ（topics）は記事の分類と発見性に直接影響する:

- **トピックページ**: `/topics/{tag}` に記事が一覧表示される
- **フォロー機能**: ユーザーはトピックをフォローでき、新着記事がフィードに表示される
- **検索流入**: Google がトピックページをインデックスする

### 5.2 タグ選定の優先度

| 優先度   | タグ例                                      | 理由                                     |
| -------- | ------------------------------------------- | ---------------------------------------- |
| **必須** | メイン技術名 (`claudecode`, `typescript`)   | 検索流入の要。フォロワーのフィードに表示 |
| **推奨** | カテゴリ (`ai`, `automation`, `devops`)     | トレンドページでの露出拡大               |
| **推奨** | フレームワーク名 (`nextjs`, `react`, `vue`) | 関連記事からの流入                       |
| **任意** | 汎用タグ (`tips`, `初心者向け`)             | 広範なリーチ（ただし競合も多い）         |

### 5.3 タグ数の最適化

- **上限**: 5個
- **推奨**: 3-5個
- **戦略**: メイン技術1 + サブ技術1 + カテゴリ1-2 + トレンドタグ0-1

### 5.4 人気タグ（2025-2026年のAI/テック系）

| タグ         | フォロワー傾向 | 備考                         |
| ------------ | -------------- | ---------------------------- |
| `ai`         | 非常に多い     | AI全般。広範だが競合も多い   |
| `openai`     | 多い           | ChatGPT、GPT-4o 関連         |
| `claudecode` | 増加中         | Claude Code 特化             |
| `mcp`        | 増加中         | Model Context Protocol       |
| `typescript` | 安定して多い   | フロントエンド〜バックエンド |
| `nextjs`     | 安定して多い   | React フレームワーク         |
| `rust`       | 増加中         | システムプログラミング       |
| `llm`        | 増加中         | 大規模言語モデル全般         |

## 6. 投稿スケジュール

### 6.1 曜日別パフォーマンス

Zenn の閲覧データ分析（220記事の実績）から、読者は**平日に技術記事を読む傾向**が明確:

| 曜日       | 閲覧傾向 | 適した投稿タイミング                     |
| ---------- | -------- | ---------------------------------------- |
| **月曜日** | 高い     | 前日夜に予約投稿 → 月曜朝にトレンド狙い  |
| **火曜日** | **最高** | 朝投稿が最も効果的                       |
| **水曜日** | **最高** | 朝投稿が最も効果的                       |
| **木曜日** | 高い     | チュートリアル系に適する                 |
| **金曜日** | やや低下 | 軽めのTips記事向き                       |
| **土曜日** | 低い     | 投稿を避ける（閲覧数が少ない）           |
| **日曜日** | やや低い | 翌週の投稿準備。長文なら日曜夜投稿も有効 |

### 6.2 時間帯別パフォーマンス

| 時間帯             | エンゲージメント | 適した内容                                 |
| ------------------ | ---------------- | ------------------------------------------ |
| **平日朝 7-9時**   | **高い**         | チュートリアル、速報。通勤時間にキャッチ   |
| **平日昼 12-13時** | **高い**         | Tips、比較記事。ランチタイム閲覧           |
| **平日夕 17-19時** | 中程度           | ディープダイブ。退勤後の学習時間           |
| **日曜夜 20-22時** | 中〜高           | 長文記事、まとめ。翌週の準備として読まれる |

### 6.3 投稿頻度の推奨

| 頻度    | 効果       | 備考                               |
| ------- | ---------- | ---------------------------------- |
| 毎日    | 最大リーチ | 品質維持が困難。熱量がある時期のみ |
| 週2-3回 | **推奨**   | 品質とリーチのバランスが最適       |
| 週1回   | 安定       | 継続性重視。長文記事向き           |
| 月2-3回 | 最低限     | これ以下だと認知が薄れる           |

### 6.4 速報記事の投稿タイミング

新ツール/ライブラリのリリース時は通常のスケジュールを無視して即座に投稿する:

- **理想**: リリースから **6時間以内** に速報記事を公開
- **許容**: リリースから **24時間以内**
- **遅すぎ**: 48時間以上経過すると他の記事に埋もれる

Source: [この1年ほぼ毎日Zenn記事を書き続けて思ったこと](https://zenn.dev/acntechjp/articles/8a4547ea504f48)

## 7. 品質基準

### 7.1 文字数の目安

| 文字数         | 評価                       | 適した記事タイプ           |
| -------------- | -------------------------- | -------------------------- |
| 1,000字未満    | パフォーマンス高い場合あり | Tips、速報、バグ修正メモ   |
| 1,000-3,000字  | やや低い                   | 短い解説には中途半端       |
| 3,000-6,000字  | **推奨**                   | チュートリアル、比較記事   |
| 6,000-10,000字 | 良い                       | ディープダイブ、完全ガイド |
| 10,000字以上   | パフォーマンス高い場合あり | 本格的な解説。読了率は低下 |

**注意**: 文字数と閲覧数の相関は非常に弱い（相関係数 -0.0223）。短い記事と長い記事の両極端が高パフォーマンスで、中程度は低パフォーマンスの傾向。

### 7.2 コードブロック

- **数**: 3個以上推奨
- **言語指定**: 必須（`typescript`, `bash`, `yaml` など）
- **実行可能性**: コピペで動作する完全版を掲載
- **コメント**: 重要な行にはインラインコメントを付ける

### 7.3 見出し構成

- **H2**: 大セクション（5-8個が目安）
- **H3**: サブセクション（各H2に1-3個）
- **目次**: Zenn は自動目次を生成するため、見出しが適切なら読者がナビゲートしやすい

### 7.4 ビジュアル要素

- **スクリーンショット**: 手順記事には必須
- **図解/ダイアグラム**: 「わかりやすい記事には必ず図がある」（トレンド入り記事の共通特徴）
- **Mermaid**: Zenn は Mermaid 記法に対応しており、コードとしてダイアグラムを埋め込める

Source: [トレンド入りする記事を書く技術](https://zenn.dev/collabostyle/articles/858875b235cdd6)

## 8. 「本」機能の活用戦略

### 8.1 本と記事の使い分け

| 比較項目   | 記事（Articles）       | 本（Books）                        |
| ---------- | ---------------------- | ---------------------------------- |
| 形式       | 単体の記事             | 複数チャプターの体系的コンテンツ   |
| 収益化     | バッジのみ             | 有料販売可能（0-5,000円）          |
| 適した内容 | 速報、Tips、単体の解説 | チュートリアルシリーズ、完全ガイド |
| SEO        | 記事単体で検索される   | チャプターごとにインデックスされる |

### 8.2 収益構造

- **価格設定**: 0-5,000円
- **最低価格**: 200円（有料の場合）
- **手数料**: 決済手数料 3.6% + プラットフォーム利用料 10% + 振込手数料 350円/回
- **著者受取額の計算例**: 1,000円の本 → 著者受取 約868円

### 8.3 収益化戦略

1. **無料記事で信頼を構築** → いいねとフォロワーを蓄積
2. **シリーズ記事を本に再構成** → 体系化した内容で有料化
3. **一部チャプターを無料公開** → 試し読みで購入を促進
4. **低価格スタート（200-500円）** → 購入ハードルを下げる

### 8.4 売上の目安

- 500円の本: 月3,000-4,000円の売上（安定期）
- 1,000-2,000円の本 × 300+購入: 年間30万-60万円の収益も可能

Source: [Zenn 本の販売や有料バッジの受取についてのよくある質問](https://zenn.dev/faq/sales), [Booksを書いて一年経ったので売上を公開します](https://zenn.dev/dirtyman/articles/3bbfe2f9eea7b1)

## 9. Publication（組織メディア）の活用

### 9.1 概要

Publication は、Zenn 上にチームのメディアを開設する機能。組織としてのブランディングに活用できる。

- **URL**: `https://zenn.dev/p/<publication_name>`
- **プラン**: Free（基本機能）/ Pro（高度な機能）

### 9.2 メンバーロール

| ロール          | 権限                                   |
| --------------- | -------------------------------------- |
| **Owner**       | 全権限（設定、メンバー管理、記事管理） |
| **Maintainer**  | メンバー管理（Owner変更は不可）        |
| **Contributor** | 記事投稿のみ                           |

### 9.3 活用メリット

- **ブランディング**: 組織の技術力をアピール
- **露出増加**: Zenn プラットフォーム上での露出が高い
- **レビュー機能**: 投稿前にチーム内でレビュー可能（品質向上）
- **統計ダッシュボード**: 閲覧数・いいね数の分析が可能
- **新規読者獲得**: 個人ブログでは難しい、組織に興味がないユーザーへのリーチ

### 9.4 GitHub 連携時の注意

Publication に記事を紐付ける場合、フロントマターに `publication_name` フィールドを追加する:

```yaml
---
title: "記事タイトル"
publication_name: "my-publication"
---
```

Source: [Zenn公式 - Publicationの使い方](https://zenn.dev/zenn/articles/how-to-use-publication), [Zenn Publication Proの機能と管理](https://zenn.dev/zenn/articles/publication-pro-features)

## 10. GitHub 連携のベストプラクティス

### 10.1 セットアップ

1. GitHub でリポジトリを作成（Public / Private どちらでも可）
2. Zenn ダッシュボードの「GitHub からのデプロイ」で連携
3. **重要**: 「Only select repositories」を選択し、連携リポジトリのみアクセス許可する
4. 同期対象ブランチを設定（デフォルト: main）

### 10.2 ディレクトリ構造

```
zenn-content/
├── articles/
│   ├── my-article-slug.md    # 記事ファイル
│   └── another-article.md
├── books/
│   └── my-book-slug/
│       ├── config.yaml        # 本の設定
│       └── chapter1.md
└── package.json              # Zenn CLI 用
```

### 10.3 制約事項

| 項目             | 制約                                                     |
| ---------------- | -------------------------------------------------------- |
| 連携リポジトリ数 | 最大2個                                                  |
| 同期対象         | 変更された `.md` ファイルのみ同期                        |
| 削除             | ダッシュボードからのみ可能（リポジトリで削除しても残る） |
| Publication連携  | Proプランのみ GitHub 連携対応                            |
| ブランチ         | 1つのブランチのみ同期対象                                |

### 10.4 Zenn CLI の活用

```bash
# インストール
npm install zenn-cli

# 初期化（ディレクトリ構造を自動作成）
npx zenn init

# 新規記事作成
npx zenn new:article --slug my-article-slug

# ローカルプレビュー
npx zenn preview

# 新規本作成
npx zenn new:book --slug my-book-slug
```

### 10.5 CI/CD 統合のヒント

- **textlint**: マークダウンの文章校正を自動化
- **GitHub Actions**: push 時に textlint を実行し、品質を担保
- **画像**: `/images/` ディレクトリに配置し、記事から相対パスで参照

## 11. AI/テック系で伸びる記事パターン

### 11.1 2025-2026年のトレンドキーワード

データ分析から、Zenn で最も高い閲覧数を獲得しているキーワード:

| キーワード               | 出現頻度（人気記事内） | 備考                                       |
| ------------------------ | ---------------------- | ------------------------------------------ |
| **Claude / Claude Code** | 最高                   | 長寿命記事のタイトルに最多出現（7回）      |
| **Copilot**              | 高い                   | AI コーディング支援の比較記事が人気        |
| **OpenAI / GPT**         | 高い                   | 新モデルリリース時にバズりやすい           |
| **MCP**                  | 増加中                 | Model Context Protocol。新しい技術への関心 |
| **Gemini**               | 中程度                 | Google AI 関連                             |
| **DeepSeek**             | 急増                   | 2025年に話題になった中国製LLM              |

### 11.2 高パフォーマンス記事の共通特徴

1. **具体的な固有名詞をタイトルに含む**: 技術名・ツール名が明示されている
2. **実際に手を動かした一次情報**: 「試してみた」「作ってみた」系
3. **シリーズ化**: 単発より連載の方が読者の追従性が高い
4. **図解・スクリーンショット入り**: ビジュアルサポートが理解を助ける
5. **環境情報が明確**: バージョン、OS、前提条件が記載されている

### 11.3 長寿命記事 vs 短寿命記事

| タイプ                 | 寿命 | 例                                                  |
| ---------------------- | ---- | --------------------------------------------------- |
| **基本的な使い方**     | 長い | 「Next.js の基本設定ガイド」                        |
| **エラー解決**         | 長い | 「TypeScript strict mode でよくあるエラーと解決法」 |
| **最新動向/速報**      | 短い | 「GPT-5 の新機能まとめ」（鮮度が命）                |
| **ベストプラクティス** | 長い | 「React Testing Library ベストプラクティス」        |

- 単一技術への集中度が高い投稿が全体の3割以上を占め、**深掘りコンテンツの需要が明確**

## 12. やってはいけないこと

### 12.1 コミュニティガイドライン違反

以下は Zenn のガイドラインで明確に禁止または非推奨とされている行為:

| 行為               | リスク         | 備考                                             |
| ------------------ | -------------- | ------------------------------------------------ |
| **AI 記事の乱造**  | アカウント制限 | 正確性を確認せずに投稿、フォロワー獲得目的の量産 |
| **広告目的の投稿** | 非公開化       | 製品宣伝や採用告知が主目的の記事                 |
| **クリックベイト** | 信頼低下       | 「衝撃！」「知らないとヤバい」等の誇張タイトル   |
| **著作権侵害**     | 削除・凍結     | 出典明記なしの引用、コードの無断転載             |
| **外部サイト誘導** | 可視性低下     | 外部サイトへ流入させることが主目的の記事         |

### 12.2 ペナルティの種類

Zenn のペナルティは複数段階で、以下のような措置がある:

- **可視性の低下**: トレンドページや検索結果からの除外（見えにくい措置）
- **記事の非公開化**: 個別記事の強制非公開
- **アカウント凍結**: 有期限または無期限
- **措置は段階的**: いきなり凍結ではなく、警告→制限→凍結の流れ

### 12.3 コンテンツ面のアンチパターン

| アンチパターン         | 問題点           | 改善策                             |
| ---------------------- | ---------------- | ---------------------------------- |
| 動作しないコード例     | 読者の信頼を失う | 必ず動作確認してから掲載           |
| バージョン情報なし     | 再現性がない     | 使用した全ツールのバージョンを明記 |
| コピペ記事             | 付加価値なし     | 独自の知見・考察を必ず追加         |
| 過度な自己アピール     | 敬遠される       | 技術的価値を最優先                 |
| 「いかがでしたか」系   | 読者離脱         | 具体的なまとめと次のステップを提示 |
| タイトルと内容の不一致 | 信頼失墜         | タイトルが約束した内容を確実に提供 |

### 12.4 AI 執筆に関するルール

Zenn は AI 執筆を**禁止していない**が、以下が求められる:

- 内容の正確性を**著者が確認**してから投稿する
- フォロワー獲得やスコア上昇を目的とした**記事の乱造は控える**
- **実体験に基づく気づきや考察**を含めることが推奨される

Source: [Zenn コミュニティガイドライン](https://zenn.dev/guideline), [AI執筆ガイドライン改定](https://www.itmedia.co.jp/aiplus/articles/2506/05/news104.html)

## 13. Zenn API の仕様と制約

### 13.1 非公式 API

Zenn は公式 API ドキュメントを公開していないが、以下のエンドポイントがコミュニティで発見されている:

| エンドポイント                                            | 用途                   |
| --------------------------------------------------------- | ---------------------- |
| `GET /api/articles?username={user}&order=latest&page={n}` | ユーザーの記事一覧取得 |
| `GET /api/articles/{slug}`                                | 記事詳細取得           |
| `GET /api/articles?topicname={topic}&order=latest`        | トピック別記事一覧     |

### 13.2 制約

- **公式サポートなし**: いつでも仕様変更される可能性がある
- **1ページ最大48件**: ページネーションで `next_page` が null になるまで取得
- **Rate Limit**: 公式には非公開。常識的な範囲（数秒間隔）で使用すること
- **認証不要**: 公開記事の取得には認証が不要

### 13.3 自動投稿の注意点

Zenn の記事投稿は API ではなく **GitHub 連携**で行う:

1. GitHub リポジトリに `.md` ファイルを push
2. Zenn が自動で同期・公開
3. 投稿頻度の制限は明記されていないが、乱造は避ける

Source: [Zenn API で記事数・いいね数を取得する](https://zenn.dev/karaage0703/articles/c24072adc188a6)

## 14. 文章スタイル

### 14.1 推奨されるトーン

- **技術的に正確**: 推測と事実を明確に区別する
- **読みやすい日本語**: 一文は60文字以内を目安に
- **敬体（です/ます）推奨**: 技術記事でもカジュアルすぎない
- **専門用語の説明**: 初出時に簡潔な説明を付ける

### 14.2 構成テンプレート

```markdown
# [技術名] の [テーマ]

## はじめに

<!-- 3行以内で何を解決するか -->

## 環境

<!-- バージョン情報、前提条件 -->

## [メインコンテンツ H2 x 3-5]

### [サブセクション H3]

## まとめ

<!-- 要点 + 次のステップ -->

## 参考リンク
```
