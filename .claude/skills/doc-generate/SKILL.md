# Documentation Generator

学習科学の原則に基づき、読者が確実に理解できる技術ドキュメントを生成するスキル。

## 発動条件

- 「ドキュメント作成」「ドキュメント生成」「docs書いて」
- 「技術文書を書いて」「解説ドキュメント」
- 新しい技術ドキュメントの作成を依頼された時

## ドキュメント種別の判定

| ユーザーの要求                         | 種別                             | テンプレート                                                      |
| -------------------------------------- | -------------------------------- | ----------------------------------------------------------------- |
| 「なぜこの技術を選んだか」「技術選定」 | **技術選定ドキュメント**         | @references/section-templates.md の「技術選定」セクション         |
| 「コードの解説」「構成を説明」         | **コードベース解説ドキュメント** | @references/section-templates.md の「コードベース解説」セクション |
| 「使い方」「セットアップ」             | **ハウツーガイド**               | @references/section-templates.md の「ハウツーガイド」セクション   |

## ワークフロー

```
[Phase 1] 調査・設計
├── 対象の理解（コード読解 / 技術調査）
├── 読者レベルの確認
└── セクション構成の設計

[Phase 2] 執筆
├── セクション構成テンプレートに従い執筆
├── 学習科学の原則を適用
└── コード例・比較表・図の作成

[Phase 3] 整合性チェック
├── 理解度チェックの整合性検証（最重要）
├── フォーマット規則の確認
└── 読み直しと推敲
```

## 学習科学の原則（必ず適用）

詳細は @references/learning-science.md を参照

| 原則                       | 適用方法                                       | 必須度 |
| -------------------------- | ---------------------------------------------- | ------ |
| **Feynman Technique**      | 1セクション = 1概念。6歳に説明するつもりで書く | 必須   |
| **アクティブリコール**     | 各セクション末尾に「理解度チェック」を配置     | 必須   |
| **スキャフォールディング** | 概要 → 詳細 → コードの順で段階的に深掘り       | 必須   |
| **チャンキング**           | 1セクション内の情報は 3〜5 個の塊に整理        | 必須   |
| **具体例 → 抽象化**        | まずコード例を見せ、そこからパターンを抽出     | 推奨   |
| **スパイラル学習**         | 同じ概念にセクションを跨いで異なる深さで再登場 | 推奨   |

## 理解度チェックの鉄則（最重要ルール）

**理解度チェックの全ての問題は、そのセクション内（チェックより前の部分）の記述だけで答えが導けなければならない。**

チェックリスト:

1. 問題を作成したら、セクション内に答えの根拠があるか逆引き確認する
2. 「〜の理由を説明できるか？」→ 本文に「なぜ」の説明があるか
3. 「〜を N つ挙げられるか？」→ 本文に N 個以上の項目が列挙されているか
4. 「〜が適しているケースは？」→ 本文にそのケースの説明があるか
5. 本文を変更したら、既存の理解度チェックに影響がないか確認する
