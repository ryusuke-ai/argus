# 学習科学の原則 — ドキュメント作成への適用ガイド

## 1. Feynman Technique（ファインマン・テクニック）

**原則**: 概念を「6歳の子どもに説明するように」シンプルに言い換える。説明できない部分が理解の穴。

**ドキュメントへの適用**:

- 1セクション = 1概念（複数概念を混ぜない）
- 全ての専門用語に初出時のインライン解説を付ける
- `> **平たく言うと**:` ボックスで非エンジニア向けの比喩を提供する

**チェック方法**: 書いた後に「エンジニアでない人に口頭で説明できるか？」を自問する。

---

## 2. アクティブリコール（能動的想起 / テスト効果）

**原則**: 情報を受動的に読むだけでなく、能動的に「思い出す」行為が長期記憶への定着を劇的に向上させる。Karpicke & Roediger (2008) の研究では、リコールを行った学生は再読のみの学生より 80% 多く記憶を保持。

**ドキュメントへの適用**:

- 各セクション末尾に「理解度チェック」を 3 問配置
- 質問の難易度を段階化する:
  - Q1: 基本確認（「〜の主な理由を 3 つ挙げられるか」）
  - Q2: 応用（「〜のメリットとデメリットを説明できるか」）
  - Q3: 深い理解（「どんなケースなら代替の方が適しているか」）

**鉄則**: 全ての問題の答えが、そのセクション内の記述から導けること。

---

## 3. スキャフォールディング（足場かけ）

**原則**: Vygotsky の「最近接発達領域（ZPD）」に基づく。学習者が独力ではできないが支援があればできるレベルの課題に対し、一時的な「足場」を提供する。

**ドキュメントへの適用**:

- 各セクション冒頭に「このセクションで学ぶこと」で学習目標を提示（事前の枠組み）
- 情報の提示順序を守る: **全体像 → 各論 → コードレベル**
- 図・表 → テキスト → コード例の順で、抽象度を段階的に下げる

---

## 4. チャンキング（情報の塊化）

**原則**: Miller のマジカルナンバー 7±2。ワーキングメモリの容量制限を考慮し、情報を意味のある小さな塊にグループ化する。

**ドキュメントへの適用**:

- 1セクション内の主要ポイントは 3〜5 個に収める
- 見出し（H2/H3/H4）で情報の階層を視覚的に示す
- リストは 3〜7 項目に整理する
- `---` 水平線でセクション間を明確に区切る

---

## 5. 具体例 → 抽象化の順序

**原則**: 具体的な事例を先に示し、そこからパターンや原則を導く。「Concreteness Fading」とも呼ばれる。

**ドキュメントへの適用**:

- まず動くコード例を見せる
- 2〜3 の異なるコンテキストの例を示す
- 共通点を抽出してパターンとして名前をつける
- **コード例 → 「学ぶべきポイント」→ パターンの一般化** の順序

---

## 6. スパイラル学習

**原則**: Jerome Bruner (1960) が提唱。同じ概念に繰り返し立ち戻りながら、毎回より深い理解を積み重ねる。

**ドキュメントへの適用**:

- セクション 2（全体概要）で軽く触れた概念を、後のセクションで深掘りする
- 「横断的パターン集」で個別セクションのパターンを統合・一般化する
- 面接 Q&A でセクション横断の知識を総合的に問う

---

## 7. ADR 形式（Architecture Decision Record）

**原則**: 技術的判断の「背景 → 選択肢 → 決定 → 結果」を構造化して記録する。

**ドキュメントへの適用**（技術選定ドキュメント限定）:

- **背景**: なぜこの判断が必要だったか
- **選択肢**: 比較表で公平に並べる
- **決定**: 採用した技術とその理由
- **結果**: メリット・デメリット、不採用技術が適するケース

---

## 8. 逆ピラミッド構造

**原則**: ジャーナリズムから。最も重要な情報を冒頭に、詳細や背景を後に配置する。

**ドキュメントへの適用**:

- 各セクションの冒頭に結論・要点を配置
- パラグラフの最初の文で要点を述べる
- 読者が途中で読むのをやめても、核心の情報は得られるように構成する

---

## 9. 二重コーディング（Dual Coding）

**原則**: 言語情報と視覚情報を組み合わせると記憶の定着率が向上する。

**ドキュメントへの適用**:

- テキスト説明 + ASCII アート図の組み合わせ
- 比較表 + コード例の組み合わせ
- フローチャート + テキスト解説の組み合わせ

---

## Diataxis フレームワーク（ドキュメント分類の基盤）

ドキュメント全体の設計にはDaniele Procida の Diataxis フレームワークを参考にする。

|            | 学習（Acquisition）         | 作業（Application）       |
| ---------- | --------------------------- | ------------------------- |
| **実践的** | Tutorials（チュートリアル） | How-to Guides（ハウツー） |
| **理論的** | Explanation（解説）         | Reference（リファレンス） |

- 技術選定ドキュメント = **Explanation** 象限
- コードベース解説 = **Explanation + Reference** のハイブリッド
- ハウツーガイド = **How-to Guides** 象限
- 1つのドキュメントに複数の象限の内容を混ぜないのが原則だが、学習用途では Explanation を主軸に Reference 要素を織り交ぜるのは許容される
