# セクション構成テンプレート

## 技術選定ドキュメント

各セクションの構造:

```
### このセクションで学ぶこと
- (箇条書き 3 項目)

### 背景（なぜ選ぶ必要があったか）
> **平たく言うと**: (非エンジニア向けアナロジー)

### 選択肢の比較
| 観点 | 選択肢A | 選択肢B | 選択肢C |
(6〜8 個の比較軸。軸名の後に（）で平易な補足)

### 決定: {選択した技術}
具体例 1: (実際のコード引用)
具体例 2: (異なるコンテキストの例)

### この選定のメリット・デメリット
**メリット**: (箇条書き)
**デメリット**: (「ただし現在の規模では〜」等の文脈補足を入れる)

### {不採用技術} が適しているケース
- (箇条書きで具体的ユースケースと理由)

### もし {代替技術} を選んでいたら（任意）
- (仮定に基づく差分分析)

### 理解度チェック
- [ ] Q1: (基本確認)
- [ ] Q2: (応用)
- [ ] Q3: (深い理解)
```

```

---

## 目次
(全セクションへのリンク)

## 1. はじめに
### このドキュメントの読み方
(使用している手法: Feynman Technique, ADR 形式, etc.)
### 前提知識
(読者に必要な知識 3 項目程度)

## 2〜N. 各セクション
(上記テンプレートに従う)

## N+1. 横断的パターン集 / 設計原則
(個別セクションのパターンを統合・一般化)

## N+2. 面接想定 Q&A
### Q{N}: 「{面接官の質問}」
**回答例**:
> (技術的に正確な回答)
**噛み砕いた説明**: (アナロジー)

## N+3. 用語集（任意）
| 用語 | 読み方 | カテゴリ | 説明 |

---
> **最終更新**: {日付}
```

## 「平たく言うと」で使うアナロジーの作り方

1. **日常の物に置き換える**: データベース → ファイルキャビネット、API → レストランの注文
2. **会社/組織に例える**: apps/ → 各部署、packages/ → 共有設備
3. **身体/家に例える**: agent-core → 心臓部、hooks → 神経系
4. **プロセスに例える**: パイプライン → 郵便局の仕分け、Code Patrol → 夜間警備員

**良いアナロジーの条件**:

- 読者の日常にあるもの
- 技術的な本質（構造や関係性）が保たれている
- 1文で伝わる長さ
