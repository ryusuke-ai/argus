## 1. アルゴリズムの仕組み（2025-2026年版）

### 1.1 トレンドランキングの計算ロジック

Qiita のトレンドは**直近で増加した LGTM 数**に基づいてランキングされる。絶対数ではなく、前回更新からの「増分」で順位が決まる。

- **更新タイミング**: 毎日 **5:00** と **17:00** の2回
- **対象記事**: 投稿から **3日以内** の記事のみ
- **ランキング枠**: デイリートレンドは約20位まで表示

Source: [Qiitaトレンドについての考察](https://qiita.com/yonex/items/b531253d0f62271367b5)

### 1.2 トレンド入りの段階

| 段階                   | 条件                             | 効果                             |
| ---------------------- | -------------------------------- | -------------------------------- |
| デイリー LGTM 集計記事 | LGTM 10以上（上位20%）           | 自動更新記事に掲載される         |
| デイリートレンド       | LGTM 集計記事掲載から約半日後    | トップページに表示、大幅な流入増 |
| ウィークリートレンド   | 日曜・木曜 5:00 に集計、上位20件 | 翌週水曜のメルマガに掲載         |
| 月間トレンド           | 毎月1日・15日の 5:00 に更新      | 長期的な権威付け                 |

**トレンド進行の実例（著者事例）:**

- 2日目 17:00: 3位でトレンド入り
- 3日目 5:00: 2位に上昇
- 3日目 17:00: 1位に到達

Source: [Qiitaでの投稿がバズる様になったので、考察する](https://terrblog.com/qiita%E3%81%A7%E3%81%AE%E6%8A%95%E7%A8%BF%E3%81%8C%E3%83%90%E3%82%BA%E3%82%8B%E6%A7%98%E3%81%AB%E3%81%AA%E3%81%A3%E3%81%9F%E3%81%AE%E3%81%A7%E3%80%81%E8%80%83%E5%AF%9F%E3%81%99%E3%82%8B/)

### 1.3 LGTM 数の分布と目標

| LGTM 数  | 全体での位置 | 意味                              |
| -------- | ------------ | --------------------------------- |
| 0        | 全体の21%    | 閲覧されていないか、反応なし      |
| 1-10     | 全体の約59%  | 一般的な記事                      |
| 10+      | **上位20%**  | デイリー集計記事に掲載可能        |
| 30-100   | 上位数%      | 専門コミュニティ内で認知          |
| **100+** | トップ層     | **バズの閾値** — 分野横断でリーチ |
| 1,000+   | 殿堂入り     | 永続的に参照される定番記事        |

- LGTM/View 比率: 平均 **3〜5%** に収束する
- LGTM が多い記事は例外なく View 数も多い

Source: [Qiita全記事分析](https://qiita.com/mtitg/items/25e3d0d75429dcfeb199), [バズ考察](https://terrblog.com/qiita%E3%81%A7%E3%81%AE%E6%8A%95%E7%A8%BF%E3%81%8C%E3%83%90%E3%82%BA%E3%82%8B%E6%A7%98%E3%81%AB%E3%81%AA%E3%81%A3%E3%81%9F%E3%81%AE%E3%81%A7%E3%80%81%E8%80%83%E5%AF%9F%E3%81%99%E3%82%8B/)

### 1.4 LGTM vs ストック

| 指標         | トレンドへの影響                 | Contribution への反映     | ユーザー行動                 |
| ------------ | -------------------------------- | ------------------------- | ---------------------------- |
| **LGTM**     | 直接影響（ランキング算出に使用） | あり（高い重み）          | 「内容が正しい・役に立った」 |
| **ストック** | 影響なし                         | あり（LGTM より低い重み） | 「後で読み返したい」         |

- Contribution = LGTM とストックの重み付け総量（公式は非公開）
- トレンドには LGTM のみが影響する。ストックはランキングメールにも影響しない

Source: [Qiita Blog - いいねがLGTMに変わります](https://blog.qiita.com/like-to-lgtm/), [LGTMとストックについて](https://qiita.com/jpwgad/items/69f49d00370a50d2d476)

### 1.5 メルマガの威力

- ウィークリートレンド上位20件が翌週水曜日のメルマガに掲載される
- メルマガ経由の流入は無視できないレベル
- メルマガ掲載 → 追加 LGTM → さらなる露出、の好循環が生まれる

Source: [やらかし記事がバズった記録](https://qiita.com/yonex/items/b531253d0f62271367b5)

## 2. プラットフォーム特性

- 日本最大の技術記事共有プラットフォーム（月間UU 約800万）
- LGTM ベースのランキングシステム
- Google 検索からの流入が主要チャネル（SEO 効果大）
- トレンド入りで大幅な流入増加（トレンド中は一定ペースで LGTM が増加し、外れると収束）
- 初心者向け AI 記事が最もバズりやすい（2025年トレンド）

## 3. コンテンツ方針

### 3.1 バズりやすい記事タイプ（6パターン）

| パターン                      | 説明                                                  | LGTM 傾向            |
| ----------------------------- | ----------------------------------------------------- | -------------------- |
| **ハンズオン/チュートリアル** | 「5歳の子供に教えるつもりで書く」。前提知識を全て疑う | 最も高い             |
| **概念深掘り**                | 抽象的な概念を共感的なメタファーで説明                | 高い                 |
| **体験談/エッセイ**           | 失敗談をユーモアで包み、具体的な教訓を抽出            | 高い（共感力で拡散） |
| **速報型**                    | 新技術発表の数時間以内に動くコードを提示              | 速度が命             |
| **まとめ/リスト**             | 主観的コメント付き（「私はこれを使っている」）        | 安定して高い         |
| **仕組み探求**                | OS/プロトコルレベルの動作原理を掘り下げ               | ニッチだが100+狙える |

Source: [Qiitaでバズるには？6つの実践パターン](https://qiita.com/makimuramakoto/items/ebd36daf5832ead10eb6)

### 3.2 2025年に伸びたジャンル

- **生成AI/LLM**: Claude Code, Copilot, Gemini, GPT-5, DeepSeek
- **AI駆動開発**: 実践的なアウトプット系記事が急増
- **ソフトスキル/マインドセット**: エンジニアとしてのキャリア論がトップ3入り

Source: [2025年技術トレンドTOP10](https://qiita.com/papasim824/items/47a221eed3ff2428663c), [Qiitaいろいろランキング2025](https://qiita.com/Qiita/items/67f7fc1c79173c9a6c44)

### 3.3 トレンド入り戦略

**初速の LGTM を稼ぐ（12-24時間で LGTM 10 が最低ライン）:**

1. X (Twitter) でシェアして初速を作る
2. フォロワーの属性に合わせた記事テーマを選ぶ
3. 投稿直後2時間が勝負（5:00 or 17:00 のトレンド更新に間に合わせる）
4. タグ5個を全て活用し、フォロワー数が多い活発なタグを選択

**理想的な投稿タイミング:**

- **朝8:00に投稿** → 9:00の集計に間に合わなくても、17:00集計でトレンド入りを狙える
- **15:00-16:00に投稿** → 17:00集計を直接狙う

Source: [バズ考察](https://terrblog.com/qiita%E3%81%A7%E3%81%AE%E6%8A%95%E7%A8%BF%E3%81%8C%E3%83%90%E3%82%BA%E3%82%8B%E6%A7%98%E3%81%AB%E3%81%AA%E3%81%A3%E3%81%9F%E3%81%AE%E3%81%A7%E3%80%81%E8%80%83%E5%AF%9F%E3%81%99%E3%82%8B/)

## 4. タイトル設計

### 4.1 基本ルール

- **20-36文字** が最適（LGTM 平均値・中央値ともに最高）
- ターゲットレベルを明示（「初心者向け」「実践」等）
- 技術名を必ず含める（検索流入のため）
- 具体的な数値や成果を入れる

### 4.2 効果的なタイトルテクニック

| テクニック     | 例                                                 | 効果                           |
| -------------- | -------------------------------------------------- | ------------------------------ |
| 【】で分類     | 【初心者向け】Claude Code で TDD を始める3ステップ | 5秒で対象読者が判断できる      |
| 親近感マーカー | 元Dockerアレルギーの私が…                          | 読者の共感を引く               |
| 具体的数値     | 10選、3ステップ、50%改善                           | 内容の具体性を示す             |
| Before/After   | 「○○をやめたら△△になった」                         | 変化のストーリーが読みたくなる |

### 4.3 良い例 / 悪い例

**良い例:**

- 「【初心者向け】Claude Code で TDD を始める3ステップ」
- 「TypeScript 5.x の新機能まとめ（実コード付き）」
- 「元Dockerアレルギーの私がコンテナ化で得た5つの教訓」

**悪い例:**

- 「最近やってみたこと」（抽象的すぎる）
- 「Claude Code について思うこと」（具体性がない）
- 「プログラミングメモ」（検索に引っかからない）

Source: [Qiita全記事分析](https://qiita.com/mtitg/items/25e3d0d75429dcfeb199), [バズらせる6個の必殺技](https://qiita.com/Ancient_Scapes/items/bba0fab1e55936e7234a)

## 5. タグ設計

### 5.1 基本ルール

- 最大5個。**全枠を使い切る**（フォロワー数が多いタグほど露出が増える）
- Qiita 形式: `[{name: "ClaudeCode", versions: []}, ...]`
- メイン技術 + 関連技術 + カテゴリの組み合わせ

### 5.2 タグ選定戦略

| 優先度 | タグ例                                | 理由                             |
| ------ | ------------------------------------- | -------------------------------- |
| 必須   | メイン技術名（ClaudeCode, Python 等） | 検索・フィード表示の基盤         |
| 推奨   | 言語/FW名（TypeScript, React 等）     | 技術スタック明示でフォロワー多い |
| 推奨   | カテゴリタグ（AI, 初心者, 入門）      | ターゲット明示                   |
| 戦略的 | 旬のタグ（生成AI, LLM, MCP）          | トレンド追い風を受ける           |
| 補完   | 汎用タグ（ハンズオン, Tips）          | 幅広い読者にリーチ               |

### 5.3 2025年のホットタグ

- AI / 生成AI / LLM / Claude / ClaudeCode
- Python / TypeScript / React
- Docker / AWS / Kubernetes
- 初心者 / 入門 / 新人プログラマ応援

Source: [Qiitaいろいろランキング2025](https://qiita.com/Qiita/items/67f7fc1c79173c9a6c44), [Qiita新規作成タグレポート](https://qiita.com/nkay/items/79bd3f1ed5dce147648f)

### 5.4 タグの SEO 効果

- タグはQiita内の検索だけでなく、Google検索にも影響する
- タグフィードのフォロワーに記事が配信される
- フォロワー数の多いタグ = 潜在的な閲覧者が多い
- タグキーワードとタイトルが一致すると検索ヒット率が上がる

Source: [エンジニアがQiitaを使うメリット](https://miraie-group.jp/sees/article/detail/engineer_Qiita_merit)

## 6. 投稿スケジュール

### 6.1 時間帯別パフォーマンス

| 時間帯               | エンゲージメント | 適した内容             | トレンド戦略                |
| -------------------- | ---------------- | ---------------------- | --------------------------- |
| **月曜朝 7-8時**     | **最も高い**     | 週初めの学習記事、力作 | 17:00集計でトレンド入り狙い |
| 平日朝 8-9時         | 高い             | チュートリアル系       | 同日17:00集計を狙う         |
| **平日昼 12-13時**   | 高い             | 昼休み読み切りサイズ   | Tips、短め記事              |
| 平日 15-16時         | 中程度           | 実践的記事             | 17:00集計直前の投稿         |
| **平日夕方 18-19時** | 高い             | 退勤後の技術記事       | 翌5:00集計を狙う            |

### 6.2 曜日別の傾向

| 曜日     | 投稿者の傾向             | LGTM の傾向            |
| -------- | ------------------------ | ---------------------- |
| **月曜** | 力作が集中（文字数多い） | **最高**（読者も集中） |
| 火-木    | 平均的                   | 安定                   |
| 金曜     | やや少ない               | 平均的                 |
| **土日** | 長い記事が多い           | **LGTM は伸びにくい**  |

**結論**: 月曜朝 7-8時が最強。土日は読者が少なく非効率。

Source: [Qiita全記事分析](https://qiita.com/mtitg/items/25e3d0d75429dcfeb199)

### 6.3 トレンド更新を意識した投稿タイミング

```
投稿 → LGTM蓄積 → 5:00 or 17:00 の集計 → トレンド入り → さらにLGTM増加
```

- 朝 **8:00** に投稿 → 17:00集計でトレンド入りを狙う（9時間の蓄積時間）
- 夕方 **15:00-16:00** に投稿 → 17:00集計に間に合わせる（短期決戦）
- 夜 **20:00-21:00** に投稿 → 翌日5:00集計を狙う（夜間の読者獲得）

Source: [Qiitaでバズるとこうなる](https://tech-blog.s-yoshiki.com/entry/184)

## 7. コンテンツフォーマット別パフォーマンス

### 7.1 記事の最適パラメータ

| パラメータ         | 最適値                          | 根拠                        |
| ------------------ | ------------------------------- | --------------------------- |
| **タイトル文字数** | 20-36文字                       | LGTM 平均・中央値が最高     |
| **本文文字数**     | 5,000-15,000文字（目安1万文字） | 1万文字まではLGTMと正の相関 |
| **セクション数**   | 8-9セクション                   | バズ記事の平均値            |
| **コード挿入回数** | 5回程度                         | バズ記事の平均値            |
| **タグ数**         | 3-5個（5個推奨）                | 露出最大化                  |

Source: [Qiita全記事分析](https://qiita.com/mtitg/items/25e3d0d75429dcfeb199), [バズ考察](https://terrblog.com/qiita%E3%81%A7%E3%81%AE%E6%8A%95%E7%A8%BF%E3%81%8C%E3%83%90%E3%82%BA%E3%82%8B%E6%A7%98%E3%81%AB%E3%81%AA%E3%81%A3%E3%81%9F%E3%81%AE%E3%81%A7%E3%80%81%E8%80%83%E5%AF%9F%E3%81%99%E3%82%8B/)

### 7.2 本文構成ガイドライン

**基本構成（チュートリアル型）:**

1. **導入** — 何が学べるか、前提知識は何か、対象読者の明示
2. **環境構築** — 最小限の手順で動く状態に
3. **本論** — コード例付きで実践的に（ステップバイステップ）
4. **応用** — 発展的な使い方、実際のユースケース
5. **まとめ** — 要点の振り返り、次のステップへの導線

**バズ記事に共通する4つの構造要素:**

1. **問題の明確化**: 読者の状況を具体的に定義・検証する
2. **思考の可視化**: 失敗した仮説も含めて過程を見せる（論理の飛躍を避ける）
3. **リスク明示**: アンチパターン、セキュリティ/効率のトレードオフも併記
4. **段階的スキャフォールディング**: 次のレベルのトピックに導く（1記事を成長経路に）

Source: [Qiitaでバズるには？6つの実践パターン](https://qiita.com/makimuramakoto/items/ebd36daf5832ead10eb6)

### 7.3 冒頭の5秒ルール

読者は最初の5秒で続きを読むか判断する。冒頭テンプレート:

1. **対象読者に直接語りかける**（「○○で困っていませんか？」）
2. **問題に共感を示す**（「私も最初は同じでした」）
3. **具体的な価値を約束する**（「この記事を読めば○○ができるようになります」）
4. **読後の変化を保証する**（「環境構築から実装まで、15分で完了します」）

Source: [Qiitaでバズるには？](https://qiita.com/makimuramakoto/items/ebd36daf5832ead10eb6)

## 8. 品質基準

- 文字数: 5,000-15,000文字（できれば1万文字程度）
- コードブロック: 実行可能な完全版を掲載
- 環境構築のみで終わらない（実用的な成果まで含める）

## 9. Organization 活用

### 9.1 Qiita Organization とは

企業・組織がメンバーの投稿を一箇所にまとめて公開できる機能（無料）。登録数は1,500以上。

### 9.2 メリット

| メリット       | 詳細                                 |
| -------------- | ------------------------------------ |
| ブランディング | 組織の技術力をアピール               |
| 可視化         | 記事数・LGTM数で組織の活動度が分かる |
| 採用効果       | エンジニアへの情報発信チャネル       |
| 管理機能       | メンバー権限の管理パネル             |

### 9.3 個人利用への示唆

- Organization に所属していなくても、Organization のメンバーとして記事を書くと露出が増える
- 所属 Organization のフォロワーにも記事が配信される

## 10. やってはいけないこと

### 10.1 コミュニティガイドライン違反

| 違反行為                     | ペナルティ                            |
| ---------------------------- | ------------------------------------- |
| 誹謗中傷・差別的表現         | 記事非公開 → 繰り返しでアカウント凍結 |
| 個人情報の無許可投稿         | 即座に非公開措置                      |
| 宣伝・販売を主目的とした記事 | 記事非公開                            |
| 著作権・肖像権侵害           | 記事削除                              |
| 不正な Contribution 獲得     | 表彰対象外 → アカウント凍結           |

### 10.2 スパム判定される行為

- **相互 LGTM の組織的実行**: 大学の講義等での相互LGTM がトレンド汚染として問題視された事例あり
- **短期間での大量投稿**: 質の低い記事の大量生産
- **タイトル詐欺**: 内容と乖離したクリックベイトタイトル
- **公式ドキュメントのコピペ**: オリジナルの知見を加えないのは NG
- **未検証コード**: 動かないコードの掲載は信頼を損なう

### 10.3 低品質記事のリスク

- Qiita は「再利用性・汎用性の高い情報が集まる場」を目指している
- エンジニアに有益でない記事は配信が抑制される可能性がある
- 繰り返し低品質な記事を投稿するとアカウント制限のリスク

### 10.4 段階的ペナルティ

1. 初回違反: 記事の非公開措置
2. 繰り返し違反: アカウントでの行動禁止
3. 重大・継続的違反: アカウント凍結

Source: [Qiitaコミュニティガイドライン](https://help.qiita.com/ja/articles/qiita-community-guideline), [Qiitaスパム記事パターン](https://gist.github.com/KEINOS/5b10b34f87955dfc86d310cd623a61d1), [Qiitaトレンド問題 Togetter](https://togetter.com/li/1647298)

| **認証済み** | 1時間あたり **1,000リクエスト**        | アクセストークン使用 |
| **未認証**   | 1時間あたり **60リクエスト**（IP単位） | 読み取り専用に近い   |

### 11.2 主要エンドポイント

### 11.4 安全な自動化の設計原則

1. **投稿のみ自動化** — LGTM やストックの自動操作は不正行為
2. **投稿間隔を十分に空ける** — 短時間の連続投稿はスパム判定リスク
3. **レートリミットを守る** — レスポンスヘッダーで残リクエスト数を確認
4. **品質を担保** — 自動投稿でも内容の正確性・有用性は必須
5. **1日1記事を上限の目安に** — 週1-2本の高品質記事が最適

### 11.5 禁止される自動化

- LGTM の自動操作（不正な Contribution 獲得）
- スパムアカウントによる LGTM 水増し
- 同一内容の反復投稿
- フォロー/アンフォローの自動操作

## 13. 文章スタイル

- **「です・ます」調** で統一
- **1文は80文字以内**
- **3項目以上は箇条書き**
- **重要ポイントは太字**
- **コードブロックには言語指定**（```typescript 等）
- **見出しは H2/H3 を使い、H1 はタイトルのみ**
- **導入に目次を含める**（長文記事の場合）

## 14. 禁止事項（記事内容）

1. 公式ドキュメントのコピペ — オリジナルの知見を加える
2. 未検証コード — 動くコードのみ掲載
3. 攻撃的トーン — 建設的な技術批判のみ
4. 過度な自己宣伝 — 技術的価値を優先
5. 「いかがでしたか」系の定型文
6. AI 生成コンテンツの無検証投稿 — 正確性を必ず確認
