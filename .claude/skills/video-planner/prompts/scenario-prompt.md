# シナリオ構成生成依頼

## 入力情報

- **テーマ**: [動画のメインテーマを記入]
- **ターゲット視聴者**: [想定視聴者像を記入]
- **動画モード**: [dialogue / narration を記入]
- **トーン**: [下記から選択]

---

## トーン選択（重要）

動画の雰囲気を決める。これによってkeyPointsの書き方が変わる。

| トーン | 特徴 | 向いているコンテンツ |
|-------|------|---------------------|
| **news** | 「へー、そうなんだ」という発見。淡々としつつ皮肉や驚きを混ぜる | デイリーニュース、アップデート情報 |
| **tutorial** | 「なるほど、こうやるのか」という理解。ステップバイステップ | ハウツー、設定ガイド |
| **deep-dive** | 「そういうことだったのか」という納得。背景や因果を掘る | 技術解説、分析レポート |
| **story** | 「それで、どうなった？」という引き込み。時系列で語る | 事例紹介、ドキュメンタリー |

**デフォルト**: news

---

## keyPointsの書き方（最重要）

### 禁止：演出指示・抽象語

keyPointsに「やること」を書くな。「伝える中身」を書け。

| ❌ NG（演出指示・抽象） | ✅ OK（具体的な中身） |
|----------------------|---------------------|
| 「挨拶」 | 「今日のニュースは情報量が多い、覚悟して」 |
| 「テーマ紹介」 | 「Claude Code開発者本人がワークフローを公開した」 |
| 「この動画で学べること」 | 「真似できる形に落とせるかが勝負」 |
| 「要点を整理」 | 「迷いを減らすのが速さの本質」 |
| 「視聴者へのメッセージ」 | 「明日から並列セッションに役割名を付けてみて」 |

### トーン別のkeyPointsの書き方

#### news（ニュース）

「何が起きた」「なぜ面白い」「視聴者への示唆」の3点セット。

```
❌ NG:
- Boris Chernyのワークフロー紹介
- 5並列セッションの説明
- まとめ

✅ OK:
- 開発者本人がワークフローを公開した。真似できる形に落とせるか？
- 5並列セッション、ただ開くだけじゃ散らかる。役割固定がポイント
- Opus一択の理由。軽いモデルで3回やり直すより、重いので1発
```

#### tutorial（チュートリアル）

「まずこれ」「次にこれ」「ここで詰まりやすい」「できたらこうなる」。

```
❌ NG:
- 手順1の説明
- 手順2の説明
- 注意点

✅ OK:
- まずClaude.mdを作る。プロジェクトの前提を書く場所
- @メンションでファイルを渡す。全部渡すな、必要なものだけ
- よくある失敗：コンテキストがパンパンで迷子になる
```

#### deep-dive（深掘り）

「そもそも何か」「なぜそうなっているか」「だから何が言えるか」。

```
❌ NG:
- AIエージェントの定義
- 特徴の解説
- 活用事例

✅ OK:
- 自律性って「次に何すればいい？」と聞かなくていいこと
- なぜ自律できる？ツールを使えるから。ファイルを読み、コードを書き、テストを走らせる
- つまり「判断は人間、作業はAI」の分担ができる
```

#### story（ストーリー）

「始まり」「転機」「どうなった」「何を学んだ」。

```
❌ NG:
- 背景説明
- 課題発生
- 解決
- 教訓

✅ OK:
- 512GBの結婚式動画が壊れた。胃が痛くなる規模感
- 魔法じゃなく手順で攻める。PhotoRecでコピー、IDR解析、FFmpegで再構築
- 復元できた。ポイントは各段で検証すること
- 困ったときほど、計画と検証が効く
```

## セクション構成のガイドライン

### オープニング（必須）

**視聴者の状態**: まだ興味が薄い。最初の10秒で「見る価値がある」と思わせる。

| やること | 具体例 |
|---------|-------|
| 今日の主役を一言で | 「開発者本人のワークフローが公開された」 |
| なぜ見る価値があるか | 「真似できる形に落とせれば、迷いが減る」 |
| 全体像をチラ見せ | 「ワークフロー、アップデート、復元事例。この3本柱で」 |

**禁止**: 「今日は〇〇について解説します」「この動画では〇〇を学べます」（教科書的）

### 本編セクション（2-5個）

**1セクション = 1つの「へー」**。視聴者が「へー、そうなんだ」と思うポイントを1つに絞る。

| セクションの型 | keyPointsの流れ |
|--------------|----------------|
| **発見型** | 意外な事実 → 理由 → だから何 |
| **比較型** | Aという常識 → でも実はB → 使い分け |
| **手順型** | まずこれ → 次にこれ → ここが落とし穴 |
| **物語型** | 困った → 試した → こうなった |

### エンディング（必須）

**視聴者の状態**: 情報を受け取った直後。「で、自分は何すればいい？」と思っている。

| やること | 具体例 |
|---------|-------|
| 今日の要点を1文で | 「速い人は、だいたい迷ってない」 |
| 明日からやれること | 「並列セッションに役割名を付けてみて」 |
| 余韻を残す | 「では、また明日」（あっさり終わる） |

**禁止**: 「いかがでしたか？」「参考になれば幸いです」（YouTuberテンプレ）

## subsection（小トピック）の考え方

1セクション内でも、**画像が変わるタイミング = subsection境界**。

```
セクション: 「5並列セッションの運用」
├── subsection: intro（概要 → 「5つ開いて並列で回す」）
├── subsection: roles（役割分担 → 「探索、実装、検証、文書化、予備」）
└── subsection: why（なぜ効く → 「コンテキスト汚染を防ぐ」）
```

## 禁止パターン（生成後にチェック）

以下がscenario.jsonに含まれていたら書き直し：

| 禁止パターン | 例 | 代わりに |
|------------|-----|---------|
| **メタ説明** | 「〇〇について解説」「〇〇を紹介」 | 具体的な中身を書く |
| **抽象名詞のみ** | 「背景」「課題」「解決」 | 何の背景か、どんな課題か |
| **動画尺の言及** | 「10分で完全理解」 | 尺に言及しない |
| **教科書的言い回し** | 「〇〇とは」「〇〇の定義」 | 口語で説明 |
| **YouTuberテンプレ** | 「いかがでしたか」「チャンネル登録」 | 使わない |

## 出力形式

```json
{
  "title": "動画タイトル（視聴者が見るもの）",
  "theme": "メインテーマ（1文）",
  "mode": "dialogue | narration",
  "tone": "news | tutorial | deep-dive | story",
  "targetAudience": "ターゲット視聴者",
  "sections": [
    {
      "id": "opening",
      "title": "内部用の詳細なセクション名",
      "displayTitle": "視聴者に見せる見出し（8-15文字）",
      "purpose": "このセクションで視聴者にどう感じてほしいか",
      "keyPoints": [
        "具体的な中身1（演出指示ではなく、伝える内容）",
        "具体的な中身2",
        "具体的な中身3"
      ],
      "visualIdeas": ["画像アイデア1", "画像アイデア2"],
      "subsections": [
        { "id": "hook", "topic": "小トピック名", "visualIdea": "画像アイデア" }
      ]
    }
  ]
}
```

### フィールド説明

| フィールド | 説明 | 注意点 |
|-----------|------|-------|
| title | 動画タイトル | 「〇分で解説」禁止 |
| displayTitle | 視聴者に見せる見出し | 8-15文字、見出しとして成立 |
| purpose | 視聴者の感情目標 | 「興味を引く」ではなく「へー、と思わせる」 |
| keyPoints | 伝える中身 | 演出指示・抽象語禁止 |
| visualIdeas | 画像のアイデア | 具体的に（「図」ではなく「5レーンの役割分担図」） |

### displayTitleの例

| NG（短すぎ/抽象的） | OK（見出しとして成立） |
|-------------------|---------------------|
| 導入 | 開発者本人の手順 |
| 全体像 | 5並列セッション |
| 性能 | Opusを選ぶ理由 |
| まとめ | 明日から試すこと |

## Few-shot例：ニューストーン

### ❌ 悪い例（堅苦しい・抽象的）

```json
{
  "id": "section-1",
  "title": "セクション1：Boris Chernyのワークフロー解説",
  "displayTitle": "ワークフロー",
  "purpose": "視聴者にワークフローの重要性を理解させる",
  "keyPoints": [
    "Boris Chernyの紹介",
    "5並列セッションの説明",
    "モデル選択の考え方",
    "Plan Modeの活用方法"
  ]
}
```
→ 問題: keyPointsが「やること」リスト。displayTitleが抽象的。

### ✅ 良い例（具体的・口語的）

```json
{
  "id": "section-1",
  "title": "トップニュース：開発者本人のワークフローを真似できる形に落とす",
  "displayTitle": "開発者の手順",
  "purpose": "「真似できそう」と思わせる",
  "keyPoints": [
    "Claude Code開発者がワークフローを公開。本人の手順は憶測より強い",
    "5並列セッション。ただ開くだけじゃ散らかる、役割固定がコツ",
    "探索、実装、検証、文書化、予備。コンテキスト汚染を防ぐ設計",
    "Opus一択の理由。軽いモデルで3回やり直すより、重いので1発が速い",
    "Plan Mode First。走る前に計画を固める。迷子防止"
  ],
  "visualIdeas": [
    "5レーンの役割分担図（探索/実装/検証/文書化/予備）",
    "Opus vs 軽量モデルの比較（ステアリング回数の観点）"
  ],
  "subsections": [
    { "id": "intro", "topic": "本人公開の価値", "visualIdea": "Boris Chernyのワークフロー紹介画像" },
    { "id": "parallel", "topic": "5並列の役割分担", "visualIdea": "5レーン図" },
    { "id": "model", "topic": "モデル選択の考え方", "visualIdea": "比較表" },
    { "id": "plan", "topic": "Plan Mode First", "visualIdea": "計画→実行のフロー" }
  ]
}
```
→ 改善点: keyPointsが「具体的な中身」。displayTitleが見出しとして成立。purposeが感情目標。
