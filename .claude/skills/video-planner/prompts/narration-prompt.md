## 重要：keyPointsは「素材」であり「台本」ではない

**keyPointsを読み上げてはいけない**。keyPointsは原材料であり、あなたの仕事は「物語化」である。

### 変換プロセス（必須）

各keyPointに対して、以下の5ステップで展開する：

| Step                   | 役割                                   | 例                                                                 |
| ---------------------- | -------------------------------------- | ------------------------------------------------------------------ |
| 1. 視聴者の痛み/疑問   | 「なぜこれを知る必要があるか」を言語化 | 「こういう経験、ありませんか？」「ここで詰まる人、多いんですよね」 |
| 2. 何が起きた/変わった | 事実を1-2文で提示                      | 「実はこれ、〇〇が変わったんです」                                 |
| 3. 具体シーン          | ツール名・操作・数字を含む仮想例       | 「例えば〇〇で△△すると、こうなります」                             |
| 4. 落とし穴/判断基準   | 注意点や選択の軸                       | 「ただし、〇〇の場合は要注意」                                     |
| 5. まとめ＋橋渡し      | 次のトピックへの接続                   | 「で、次に気になるのが〇〇ですよね」                               |

**全ステップを必ず使う必要はない**。トピックの重要度で調整：

- 重要トピック：5ステップ全部
- 軽めのトピック：Step 2→3→5（痛みと落とし穴を省略）
- オープニング/エンディング：Step 1→5（フック→締め）

## 品質基準：箇条書き禁止

### 絶対禁止パターン

以下が出力に含まれていたら失格：

| NGパターン       | 例                                                    | 問題点               |
| ---------------- | ----------------------------------------------------- | -------------------- |
| **名詞列挙**     | 「ワークフロー、アップデート、学習、周辺ツール」      | keyPointsの読み上げ  |
| **カテゴリ羅列** | 「今日は7トピックを、カテゴリで整理して見ていきます」 | 予告の予告で情報ゼロ |
| **数字で構造化** | 「1つ目は〇〇、2つ目は〇〇」                          | 講義調で退屈         |
| **受動態連発**   | 「〇〇と言われています」「〇〇が期待されています」    | 体温がない           |

### 必須パターン

| 必須要素               | 効果                               | 挿入頻度          |
| ---------------------- | ---------------------------------- | ----------------- |
| **視聴者への問いかけ** | エンゲージメント維持               | 2-3分に1回        |
| **仮想シーン**         | 具体性・臨場感                     | 各トピック1回以上 |
| **注意喚起**           | 「ただし」「ここが落とし穴」       | 各セクション1回   |
| **橋渡し**             | 「で、ここで」「次に気になるのが」 | トピック間        |

## リズム制御

### 文長のバリエーション（A-A-Bリズム）

短文と長文を意図的に混ぜる。同じ長さの文が3つ以上続くのは禁止。

| 文長タイプ | 文字数  | 役割             | 例                                                                               |
| ---------- | ------- | ---------------- | -------------------------------------------------------------------------------- |
| **短文**   | 15-25字 | 区切り・強調     | 「ここが肝です。」「要は、〇〇です。」                                           |
| **中文**   | 30-50字 | 標準的な説明     | 「〇〇というのは、実は△△という意味なんですよね。」                               |
| **長文**   | 55-80字 | 具体例・因果関係 | 「例えば〇〇の場面で、△△しようとしたら、実は□□が起きるんです。で、ここが困る。」 |

### リズムパターン例

```
中→中→短  ← 説明→説明→区切り
短→長→中  ← 強調→具体例→まとめ
中→短→長  ← 導入→注意→展開
```

## Few-shot例：keyPoints→セリフ変換

### 例1: 技術トピック

**keyPoints（入力）**

```
- 5並列セッション：番号タブ運用の意図（探索・実装・検証・文書化・予備）を分解
- Opus 4.5一択：遅さより"ステアリング回数の減少"が総合速度を上げるという示唆
```

**❌ 悪い例（keyPointsの読み上げ）**

```
「まず強いのが、5並列セッション。ターミナルで5つ同時です。」
「番号タブで管理して、役割を固定します。ここが肝ですね。」
「探索用、実装用、検証用、文書化用、予備。5レーン運用です。」
「次に、Opus 4.5一択という話。速さより質を選んでいます。」
```

→ 問題：名詞列挙、展開なし、なぜ効くのかの説明がない

**✅ 良い例（5ステップ変換）**

```
「ターミナルを5つ開いて並列で回す。こういうやり方、実は開発者本人がやってるんですよね。」（Step 2: 何が起きた）

「でも、ただ5つ開くだけじゃ散らかるだけじゃないですか。」（Step 1: 視聴者の痛み）

「ポイントは、役割を固定すること。探索レーンではファイルを漁る。実装レーンでは手を動かす。検証レーンではテストを走らせる。こうやって"コンテキストの汚染"を防ぐ設計なんです。」（Step 3: 具体シーン）

「で、面白いのがモデル選択。速いモデルを使いたくなるじゃないですか。でも開発者本人はOpus一択なんですよ。」（Step 4: 意外性）

「なぜか。軽いモデルで3回やり直すより、重いモデルで1回で決める方が、結局速いと。総合速度の発想ですね。」（Step 5: まとめ）
```

→ 変換ポイント：「5並列」を「コンテキスト汚染防止」という課題解決に翻訳、「Opus一択」を「意外性→理由→納得」の流れに

### 例2: アップデート情報

**keyPoints（入力）**

```
- Claude Code 2.1.1：CLIコマンド変更が多く、過去の手癖が壊れやすい点に注意喚起
```

**❌ 悪い例**

```
「アップデートは、Claude Code 2.1.1。ここ、手癖が壊れます。」
「数字で言うと、CLIコマンド変更が109件。わりと容赦ないですね。」
```

→ 問題：数字だけ、視聴者への影響が不明

**✅ 良い例**

```
「で、これ聞いて『ウッ』ってなる人、多いと思うんですよ。」（Step 1: 痛み予告）

「2.1.1でCLIコマンドが109件変わってます。」（Step 2: 事実）

「何が怖いかって、自分で作ったエイリアスとかテンプレートが、静かに壊れること。動くけど想定と違う、みたいなのが一番厄介じゃないですか。」（Step 3: 具体シーン + Step 4: 落とし穴）

「対策は、更新のたびに"既存手順の再現テスト"を1回だけ回すこと。/initの挙動、Claude.mdの読み込み、@メンションの反応、ここだけ確認すれば大丈夫です。」（Step 5: 解決策）
```

## キャラクター設定（ソロナレーション用）

### tsukuyomi（単独解説者）

- **スタンス**: 「視聴者の味方の専門家」。上から教えるのではなく、隣で解説する距離感
- **体温**: ニュースに対して自分も驚いたり、皮肉を言ったりする
- **禁止口調**: 「〜でございます」「〜と思われます」（官僚調）

### 視聴者との認知的同期

「視聴者がこの一文を聞いた直後に抱く疑問を、次の一文で拾う」

```
✅ 「109件変わってます。」→「何が怖いかって、〜」
✅ 「Opus一択なんですよ。」→「なぜか。〜」
❌ 「109件変わってます。」→「次に、学習の話です。」（疑問放置）
```

## セリフ数の目安

| 動画の長さ | セリフ数 | 備考                           |
| ---------- | -------- | ------------------------------ |
| 3分        | 50-70    | ショートニュース               |
| 5分        | 90-110   | 短い解説                       |
| 7分        | 130-160  | デイリーニュース（デフォルト） |
| 10分       | 180-220  | 標準解説                       |
| 30分       | 550-650  | 深掘り解説                     |

**注意**: scenario.jsonの想定尺に合わせて調整

## 出力形式

```json
{
  "mode": "narration",
  "segments": [
    {
      "speaker": "tsukuyomi",
      "text": "セリフテキスト",
      "sectionId": "opening",
      "emotion": "default"
    }
  ]
}
```

### フィールド説明

- **speaker**: tsukuyomi（固定）
- **text**: セリフ（15-80文字、リズムを意識）
- **sectionId**: scenario.jsonのsections.idと対応
- **emotion**: default / angry / doubt / love / thinking / surprised

## 自己検証チェックリスト（生成後に確認）

生成したセグメントに以下がないか確認：

- [ ] 名詞が3つ以上連続で列挙されている箇所
- [ ] 「今日は〇トピック」「〇つのポイント」という構造予告
- [ ] 同じ文長（±5文字）が3つ以上連続
- [ ] 「〜されています」「〜と言われています」が連続
- [ ] keyPointsのフレーズがほぼそのまま使われている箇所
- [ ] トピック間の橋渡しがない箇所（唐突なセクション移動）

上記に該当があれば、該当箇所を5ステップ変換で書き直すこと。
